Shiori 存档系统详细说明

← 返回主页

Shiori 引擎提供了一套完整的存档与导航系统，包含五个核心模块。这些模块采用模块化设计，既可独立运行，又能通过统一的数据源实现状态同步，为开发者提供了灵活的游戏进度管理方案。

1. 进度管理（saves.html）

功能定位

最原始的存档管理系统，提供基础的API接口支持，是整个存档体系的数据基石。

技术实现细节

  • 数据存储：通过 localStorage.gameProgress 存储游戏进度数据
  • 场景追踪：使用 sceneMarkers 数组记录玩家已访问的所有场景节点
  • 展示方式：简单的列表式展示，无复杂的UI交互和视觉效果

数据结构示例

{
  sceneMarkers: ["scene1", "scene2", "scene3"],  // 已访问场景列表
  currentScene: "scene5",                         // 当前所在场景
  currentLine: 42,                                // 当前剧情行号
  timestamp: 1716000000000                        // 最后更新时间戳
}

适用场景与局限性

优势：
  • 轻量级实现，性能开销极小
  • 作为其他高级功能的底层数据源
  • 为成就系统、统计面板提供原始数据支持
  • 开发者调试时快速查看游戏进度状态
局限：
  • 无法直接进行存档/读档操作
  • 缺乏用户友好的界面交互
  • 不支持多存档位管理
  • 无视觉反馈和状态提示

2. 小故事鉴赏（story.html）

功能定位

独立的小故事/番外篇鉴赏模块，用于展示与主线剧情解耦的额外内容。

技术实现细节

  • 内容管理：完全硬编码实现，未接入游戏进度API
  • 展示机制：静态内容展示，不支持动态解锁机制
  • 架构设计：独立的HTML结构，与主线剧情完全解耦
  • 数据隔离：不读取也不写入 localStorage.gameProgress

适用场景

  • 角色番外故事：展示角色的背景故事、个人线剧情
  • 背景设定集：世界观介绍、角色档案、术语解释
  • DLC内容入口：付费或免费追加内容的独立展示平台
  • 隐藏剧情查阅：达成特定条件后解锁的特殊剧情回顾
  • 二创作品收录：官方认可的玩家创作内容展示区

3. 行级存档恢复（archive.html）

功能定位

精细化的存档管理系统，支持精确到剧情行的保存与恢复，是玩家最常用的核心存档功能。

核心功能

  1. 完整上下文保存：
    • 当前场景文件名
    • 精确的剧情行号
    • 所有游戏变量状态（好感度、标志位、物品等）
    • 保存时间和自定义名称
  2. 多存档位管理：
    • 支持无限数量的存档（受限于浏览器存储容量）
    • 每个存档位独立存储，互不干扰
    • 支持双击编辑存档名称
  3. 智能返回机制：
    • 从存档页返回时自动定位到原剧情位置
    • 显示"返回当前页面"按钮并高亮当前节点
    • 保持流畅的游戏体验，避免迷失方向
  4. 数据导入导出：
    • 支持将所有存档导出为 JSON 文件
    • 可从 JSON 文件导入存档数据
    • 便于备份、分享和跨设备迁移

适用场景

  • 关键选择前存档：在重要分支点前手动创建存档点
  • 分支路线测试：快速切换不同选择路线进行对比
  • 困难章节挑战：在高难度段落反复尝试直至通关
  • 多结局收集：回溯到关键点探索不同结局
  • 进度备份：定期导出存档防止数据丢失

4. 剧本流程图（flowchart.html）- 正式版

功能定位

可视化的剧本导航系统，根据玩家实际游戏进度动态解锁节点，提供沉浸式的剧情回顾体验。

技术实现细节

解锁判断逻辑

// 从 saves.html 读取已访问场景
const visitedScenes = localStorage.gameProgress.sceneMarkers || [];

// 判断节点是否解锁
function isNodeUnlocked(sceneId) {
  return visitedScenes.includes(sceneId);
}

// 渲染节点状态
if (isNodeUnlocked(node.id)) {
  renderNormalNode(node);      // 正常显示，可点击
} else {
  renderLockedNode(node);      // 显示"？？？"，不可点击
}

自动布局引擎

  1. 垂直位置计算：根据节点 ID 的层级深度确定 Y 轴位置
  2. 水平分布调整：使用 x/y 百分比偏移微调节点位置，避免节点重叠和连线交叉
  3. SVG 正交连线：智能绘制节点间的关系路径，采用直角折线提升可读性

智能导航功能

  • 当前位置高亮：从剧情页进入时，高亮显示当前所在节点
  • 一键返回：提供"返回当前页面"按钮，点击后恢复到原剧情行
  • 路径追溯：可通过点击已解锁节点快速跳转到任意历史场景

5. 剧本流程图（flowchart_debug.html）- 调试版

功能定位

开发调试专用的完整流程图查看器，无视游戏进度限制，始终显示所有节点和连线。

技术实现细节

硬编码场景数据

// 调试版直接使用完整的场景定义
const allScenes = [
  { id: "scene1", label: "初次相遇", connections: ["scene2", "scene3"] },
  { id: "scene2", label: "图书馆约会", connections: ["scene4"] },
  { id: "scene3", label: "咖啡厅闲聊", connections: ["scene4"] },
  { id: "scene4", label: "告白场景", connections: ["ending_good", "ending_bad"] },
  // ... 所有场景，包括测试分支和调试标记点

// 移除解锁判断，所有节点始终可见
function renderNode(node) {
  renderNormalNode(node);  // 永远以正常状态渲染
}

系统架构总览

模块化设计原则

1. 独立运行

每个系统均可单独使用，无需依赖其他模块：

  • 可仅使用 archive.html 而不启用 flowchart.html
  • 可在项目中删除 story.html 而不影响其他功能
  • 各模块间无硬性耦合，便于按需裁剪

2. 数据互通

共享 localStorage.gameProgress 作为统一数据源：

// 统一的数据结构
localStorage.gameProgress = {
  sceneMarkers: ["scene1", "scene2"],  // saves.html 写入，flowchart.html 读取
  currentScene: "scene5",               // archive.html 更新，flowchart.html 高亮
  currentLine: 42,                      // archive.html 保存，用于智能返回
  archives: [...],                      // archive.html 独占管理
  lastUpdate: 1716000000000             // 所有模块共同维护
};

3. 按需配置

可根据项目需求灵活增减功能模块：

  • 小型项目：仅保留 archive.html 即可满足基本需求
  • 中型项目：增加 flowchart.html 提升导航体验
  • 大型项目：启用全部模块，构建完整的进度管理体系
  • 调试阶段：使用 flowchart_debug.html 加速开发迭代

4. 首次编译友好

所有新增功能均为纯前端实现：

  • 不引入额外依赖库（无 npm install）
  • 不修改核心引擎代码（engine.js 保持不变）
  • 不影响首次编译所需的文件结构
  • 编译流程与原有项目完全兼容

存档隔离机制

调试版与正式版的数据隔离

隔离原理

  • 正式版（Shiori.exe）：使用标准 WebView2 用户数据目录
  • 调试版（Shiori_debug.exe）：使用独立的调试版用户数据目录
  • 两者的 localStorage 完全隔离，互不干扰

实际应用价值

  1. 开发阶段：
    • 可在调试版中自由测试新功能
    • 不会影响正式版中玩家的真实存档
    • 可随时重置调试版数据而不担心误删玩家进度
  2. 发布前检查：
    • 在打包分发正式版前，清除调试过程中产生的测试存档
    • 通过窗口菜单的 调试 → 清除所有存档 一键清理
    • 确保正式发布版本的存档数据干净整洁
  3. 数据安全：
    • 开发者测试环境与玩家生产环境完全独立
    • 避免测试数据污染正式发布版本
    • 降低因调试操作导致玩家数据丢失的风险

清除存档功能

调试版提供一键清除所有存档的功能：

  • 位置：窗口菜单 → 调试 → 清除所有存档
  • 确认机制：执行前弹出二次确认对话框，防止误操作
  • 清除范围：
    • 所有剧本的进度存档（sceneMarkers）
    • 所有手动创建的存档位（archives）
    • 选择记录和剧情标志位（flags）
    • 好感度数据（affinity）

常见问题解答

Q1: 为什么需要这么多存档系统？

A: 每个系统针对不同场景优化：saves.html 是底层数据接口；story.html 用于独立内容展示；archive.html 是玩家日常使用的核心存档功能；flowchart.html 提供可视化导航；flowchart_debug.html 是开发工具。

Q2: 存档数据会丢失吗？

A: 存档存储在浏览器的 localStorage 中。正常情况下数据持久保存，但可能在用户手动清除浏览器数据、卸载重装游戏或存储空间不足时丢失。建议定期使用 archive.html 的导出功能备份存档。

Q3: 如何在不同设备间转移存档？

A: 使用 archive.html 的导入导出功能：在旧设备上点击"导出所有存档"保存 JSON 文件，将文件复制到新设备后点击"导入存档"选择该文件即可。

Q4: 调试版和正式版可以共存吗？

A: 可以，且推荐共存。两者使用不同的可执行文件名，存档数据完全隔离，可同时安装在同一目录下。开发者日常使用调试版，玩家使用正式版。

Q5: 流程图节点太多导致混乱怎么办？

A: 建议合理分组相关场景、使用偏移量微调节点位置、考虑未来扩展折叠/展开功能和颜色区分不同类型节点。可在 flowchart_debug.html 中预览完整布局后再调整。

最佳实践建议

对于开发者

  1. 开发阶段：
    • 始终使用调试版（Shiori_debug.exe）进行测试
    • 定期查看 flowchart_debug.html 确认剧本结构
    • 利用 F1 调试面板实时监控游戏状态
  2. 测试阶段：
    • 覆盖所有分支路线，确保无死胡同
    • 验证存档/读档功能的稳定性
    • 测试极端情况（大量存档、快速切换场景等）
  3. 发布前：
    • 清除调试版中的所有测试存档
    • 检查正式版中流程图节点的解锁逻辑
    • 确认存档导出/导入功能正常工作

对于玩家

  1. 定期备份：每完成一个重要章节后导出存档，将 JSON 文件保存在安全位置
  2. 合理使用多存档位：在关键选择前创建多个存档点，为不同路线保留独立存档
  3. 利用流程图导航：通过 flowchart.html 查看已探索的剧情，发现遗漏的场景并及时补全
